Lesson 02 · Pointers

Pointers to Structs

A pointer is just a number that names a location in memory. When that memory holds a struct, the pointer gives the callee a handle back to the caller's data — copy-free, mutation-capable.

A pointer is a variable whose value is the address of another variable. In Go a pointer has a type — you can't make a *User point at an int.

type User struct { Name string; Age int }

var u  User     // u       has type User,    value  {Name:"" Age:0}
var p  *User    // p       has type *User,   value  nil
p = &u              // &u      has type *User,   value  address-of-u
*p = User{Name: "Naveen", Age: 42}  // write through p
fmt.Println(u.Name) // "Naveen" — same memory

The two operators, side by side

Memory: pointer adds a heap, leaves the value alone

Taking the address of a local variable usually makes it escape to the heap. The Go compiler proves this at compile time: if a pointer to u could outlive main, u can't live on the stack anymore. It moves to the heap, and the garbage collector takes over.

The stack frame holds only the pointer (8 bytes). The struct lives on the heap (here, 32 bytes including padding). Reading *p means "follow the arrow".

Why does this matter? Every struct value you pass through a channel, store in a slice beyond a small size, or return from a function whose address you keep, ends up on the heap. Pointer-heavy code is heap-heavy code.

Auto-dereference: Go's best ergonomic gift

In C, (*p).Field is mandatory. In Go, the compiler does it for you:

var p *User = &User{Name: "Naveen", Age: 42}
fmt.Println(*p.Name)     // syntax error — parses as *(p.Name)
fmt.Println((*p).Name)   // fine, but ugly
fmt.Println(p.Name)        // fine, idiomatic — Go auto-dereferences
p.Age = 43            // ditto for assignment

Field access and method calls are the only two operations Go auto-dereferences for. Arithmetic, indexing, and range loops on pointers do not auto-dereference. The grammar in the Selectors section of the spec encodes this exactly.

Three ways to get a *T

type T struct { X int }

// (1) address-of a literal — most common
p1 := &T{X: 1}

// (2) address-of an existing value
v  := T{X: 2}
p2 := &v

// (3) new — zero-valued, on the heap
p3 := new(T)  // *T pointing to {X: 0}

&T{X: 1}

new(T) and &T{} produce the same shape, but new is rarely the right choice — most Go code prefers &T{...} so the initial values are visible at the call site.

The nil pointer trap

Dereferencing a nil pointer panics. There is no automatic "null object" or "default value" — a nil *T is just an address of zero, and reading from address zero is a runtime error. 
var p *User           // p == nil
fmt.Println(p.Name)   // panic: runtime error: invalid memory address

// guard with an explicit check
if p != nil {
    fmt.Println(p.Name)
}

Constructors in idiomatic Go return (*T, error) so the caller can never silently hold a nil:

func NewUser(name string) (*User, error) {
    if name == "" {
        return nil, errors.New("empty name")
    }
    return &User{Name: name}, nil
}

Putting it together: the same struct, three places

func main() {
    a := User{Name: "A"}    // stack value
    b := &User{Name: "B"}  // heap value, pointer on stack
    c := new(User)         // heap value, zeroed, pointer on stack
    c.Name = "C"

    fmt.Println(a.Name, b.Name, c.Name) // A B C
    b.Name = "B!"               // mutation via pointer
}
