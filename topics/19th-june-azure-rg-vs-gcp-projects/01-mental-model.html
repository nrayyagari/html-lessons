Lesson 01 · Foundations

The mental model: containers for cloud resources

Azure and GCP both let you group cloud resources into named containers, but the two systems come at it from opposite directions: one is flat, one is nested. Once you see the shape, the rest of the differences fall out.

What problem are they solving?

You have a subscription (Azure) or a billing account (GCP). Inside it, you can spin up dozens, hundreds, or thousands of resources — VMs, databases, storage accounts, functions. Without grouping, you have an unsorted list. Grouping solves four problems at once:

Billing boundary

Sum the cost of everything in a group. Tag it to a team, project, or environment. Show me what this costs without grepping the whole bill.

Access boundary

Grant "read everything in this group" to a team without giving them access to the rest of the subscription. The group is the unit of least-privilege.

Lifecycle boundary

Delete the group, delete everything in it. Useful for ephemeral environments, dev sandboxes, decommissioned services.

Policy / governance boundary

Apply a policy (region constraints, allowed SKUs, required tags) to the group and have it enforced for every resource inside.

Both clouds converged on this idea. The names are different, the shape of the hierarchy is different, but the four problems are the same.

The naming, side by side

Azure

Resource Group (RG)

A flat container that lives directly inside a subscription. One level of grouping. Resources belong to exactly one RG. An RG can't contain another RG.

Naming rule: alphanumerics, periods, hyphens, underscores, and parentheses. 1–90 chars. Must be unique within the subscription.

GCP

Project

A container that lives inside a folder, which lives inside an organization. Nested — up to ~8 levels of folders. Resources belong to exactly one project, but projects themselves can be organized in a tree.

Naming rule: 6–30 chars, must start with a lowercase letter, may contain lowercase letters, numbers, and hyphens.

The single most important fact. In Azure, the RG is the lowest unit of grouping and it's flat. In GCP, the project is the lowest unit of grouping and it's nested. This is the difference that drives every other difference — IAM, billing, policy, and quotas all inherit differently because the shape is different.

What "containment" actually means

When you create a VM, you tell Azure which RG it goes in. The RG is recorded in the resource's id field, which is immutable for the life of the resource:

// Azure resource ID
/subscriptions/00000000-0000-0000-0000-000000000000/
    resourceGroups/my-app-prod-rg/
    providers/Microsoft.Compute/
    virtualMachines/my-vm

You can't move a resource to a different RG. To "move" it, you create a new resource in the target RG and delete the old one. (Some resource types support actual move operations, but most don't.)

GCP projects are similar — the project is baked into every resource's identity and is not mutable. To migrate, you redeploy.

Resources don't migrate between groups. The immutability of the containing group means that "restructuring" your Azure subscription or GCP org tree is a real migration project — not a drag-and-drop in the portal. Plan the group structure up front, before you have a thousand resources to move.

The four boundary problems, mapped

ProblemAzure unitGCP unit
BillingRG (cost analyzer can scope to RG)Project (billing export per project)
Access (least-privilege)RG (RBAC scope)Project (IAM binding scope)
Lifecycle (delete all)RG (delete RG → all resources gone)Project (delete project → all resources gone)
Policy / governanceRG (Azure Policy assignment)Project (Org Policy inheritance)

Note how the unit is the same on both sides: a single project in GCP does what a single RG does in Azure. The difference is what happens above that unit — which is lesson 2.

A mental model that travels

Useful when you're switching between clouds: think of the Azure RG as the smallest grouping unit you actually use, and of the GCP project as the same, with the option to nest them. If you imagine a one-level-only GCP project, you've imagined an Azure RG.

Then in lesson 2 we'll see what the extra nesting levels in GCP buy you — and why some teams recreate the hierarchy with Azure Management Groups (a separate, optional feature most teams skip).
