Lesson 02 · Hierarchy

Flat vs nested: the structural difference

Azure's grouping is one level deep. GCP's is a tree. The shape of the hierarchy changes how IAM, billing, and policy inherit — and that is what makes multi-team, multi-environment setups work differently on the two clouds.

Once you have more than a handful of groups, you need a way to organize the groups themselves. The two clouds picked opposite answers: Azure leaves it to a separate, optional feature (Management Groups). GCP bakes a tree into the resource model.

Azure: flat + Management Groups (optional)

Inside a subscription, every Resource Group is a peer. To get hierarchy, you create a Management Group tree that lives above subscriptions:

Tenant Root Group (auto) one per Azure AD tenant
MG: "Platform"
MG: "Engineering"
MG: "Data"
Sub: prod-payments
Sub: prod-search
Sub: data-warehouse
RG: app
RG: data
RG: net
RG: jobs

Notice: RGs (the lowest level) are flat siblings inside their subscription. Hierarchy is layered on top via Management Groups, which most small teams never create.

Why so many teams skip Management Groups. If you have one subscription, or one subscription per environment (prod / stage / dev), the MG layer is redundant — the subscription boundary already does the job. MGs shine when you have 5+ subscriptions that need shared governance (e.g. "all prod subscriptions must deploy only to these 3 regions").

GCP: projects live in a folder tree, by default

Every GCP project is created inside a parent. The default parent for new orgs is the organization root, but you can nest projects under folders up to ~8 levels deep:

Organization one per Workspace
Folder: "Engineering"
Folder: "Payments"
Folder: "Search"
Project: payments-prod
Project: payments-dev
Project: search-prod

Compared to Azure, the project (the smallest unit) is the leaf of a tree, not a flat peer. The folders let you group projects by team, by environment, by business unit, by anything — without inventing a new layer like MGs.

GCP projects can also live directly under the Org. A project doesn't have to be inside a folder. You can create "orphans" directly under the organization root, which is the default for new projects in small orgs. Folders are purely organizational sugar.

What the extra levels buy you

Hierarchy is most useful for two things: IAM inheritance and billing rollup. Let's see both.

IAM inheritance

A role granted at the Org level is inherited by every folder, project, and resource under it. Grant a role at the "Engineering" folder and every project in it picks it up. Grant it at the Org root and every project in the org picks it up.

Azure works the same way at the MG and subscription levels — the same inheritance rules apply, you just have to opt into the extra levels. The mental model is identical; the difference is whether the levels are pre-baked (GCP) or you-build-them (Azure).

Billing rollup

GCP billing accounts are a separate hierarchy from folders. A project belongs to exactly one billing account; a billing account can have many projects. Folders are for organization, billing accounts are for money. They're independent axes that you wire together.

Azure has a similar split: subscriptions are both a billing boundary and an IAM boundary. You don't have to choose between them — they're the same unit.

When you switch clouds, watch this. On Azure, the subscription is both "where it bills from" and "where IAM is scoped." On GCP, those are independent. A common porting mistake is to assume a GCP project maps 1:1 to an Azure subscription — it's usually a better match for an Azure RG + subscription, or for multiple RGs inside one subscription.

Practical layout patterns

Azure: subscription-per-environment

One subscription for prod, one for nonprod. RGs inside each subscription are per application. IAM is per-RG, billing is per-subscription, delete a nonprod subscription to clean up an environment.

GCP: folder-per-environment

One folder for prod, one for nonprod, each containing the relevant projects. IAM is per-project (or per-folder if you want inheritance). Billing is per-project against one billing account, with budget alerts per folder.

Same outcome, different mechanism. Both work; pick the one that matches your team's mental model.
