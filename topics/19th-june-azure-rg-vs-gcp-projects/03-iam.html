Lesson 03 · IAM

IAM: role assignments, scope, and inheritance

Both clouds let you grant a principal a role at a scope. Azure calls the scope a "role assignment"; GCP calls it a "policy binding". Same idea, different vocabulary, one sharp difference in how the default behaves.

IAM has three moving parts on every cloud:

  1. Principal — who: user, group, service account, managed identity, workload identity.
  2. Role — what: a named bundle of permissions (e.g. Storage Object Reader, Virtual Machine Contributor).
  3. Scope — where: the boundary the role applies within (Azure RG, GCP project, or higher).

Granting is always (principal, role, scope). The difference between Azure and GCP is mostly vocabulary, the shape of the scope, and one critical default: how roles combine.

Azure: RBAC role assignments

You write a role assignment as a separate resource. The scope is explicit:

apiVersion: authorization.rbac.authorization.k8s.io/v1
kind: RoleAssignment        # usually created via portal/CLI/Terraform, not raw YAML
scope:
  /subscriptions/00000000-…/resourceGroups/payments-prod-rg
properties:
  principalId: <object-id-of-the-user-or-group>
  roleDefinitionId: <guid-of-the-role>
  # e.g. b24988ac-6180-42a0-ab88-20f7382dd24c == "Contributor"

Key properties of Azure RBAC:

Roles are additive

If a user has Reader at the subscription and Contributor at an RG, they have both on the RG. There's no "deny" except via Azure Blueprints Deny assignments or explicit deny rules — which are separate constructs.

Roles are denormalized

There's no inheritance optimization. The "Reader at subscription" role isn't "inherited" at the RG level — it's re-evaluated at every API call. The effect is the same, but the implementation is per-call, not per-resource.

Built-in vs custom roles

Azure ships ~200 built-in roles (Reader, Contributor, Owner, plus per-service). Custom roles are JSON definitions of permission sets you assemble yourself.

GCP: IAM policy bindings

In GCP, the IAM policy is a single object attached to a resource. You modify it with gcloud or Terraform — you don't create a separate "RoleAssignment" object:

$ gcloud projects add-iam-policy-binding payments-prod \
    --member="user:alice@example.com" \
    --role="roles/storage.objectViewer"

The same effect, expressed declaratively as part of the project's policy:

# terraform
resource "google_project_iam_binding" "alice_readonly" {
  project = "payments-prod"
  role    = "roles/storage.objectViewer"
  members = [
    "user:alice@example.com",
  ]
}

Key properties of GCP IAM:

Roles are additive (same as Azure)

Multiple bindings at any level combine. No deny in the default policy model (deny requires the separate Access Context Manager / Access Transparency features).

Inheritance is real

If alice has a role on the Org, it propagates to every folder, project, and resource under it. The effective IAM is the union of all bindings from the root down to the resource being checked.

Roles are pre-built

GCP has ~2500 predefined roles (per-service, per-resource). Custom roles are JSON, but the convention is to use predefined ones unless you really need custom.

The default that bites people: default deny

Both clouds deny everything by default. The difference is when that default is computed.

Azure: explicit at every scope

There's no "I get access by being in the right AAD group." You need a role assignment for each (principal, role, scope) tuple. If you want a user to read an RG, you create a role assignment. The portal has "Access control (IAM)" pages to make this easy, but it's still a per-scope grant.

GCP: inheritance helps the team model

Grant roles/viewer to an AAD/Google group at the Engineering folder once, and every project under it inherits the binding. The team doesn't have to re-grant per-project.

Azure inheritance works the same way at the subscription level (via "Access control (IAM)" on the subscription), but most teams manage RGs as the unit of access, which means more role assignments in total. The flat hierarchy forces you to be explicit; the nested hierarchy lets you be lazy once and inherit forever.

Service accounts: where the clouds really differ

Service accounts are the "give my code a credential" feature. The shape is different enough to trip people up:

ConceptAzureGCP
The principal typeManaged Identity (system-assigned or user-assigned)Service Account (an identity, not a thing attached to one resource)
Created whereAuto-created when you enable it on a resource, or explicit in the resource's identity bladeExplicit: gcloud iam service-accounts create or Terraform
How workloads authenticateIMDS endpoint returns an OAuth token — no secret to manageEither: (a) attach a JSON key file (legacy), (b) Workload Identity Federation with k8s, or (c) metadata server in GCP VMs
Cross-cloudFederate with Entra ID — AAD service principal in another tenantWorkload Identity Federation with any OIDC provider (GitHub Actions, k8s, AWS, Azure)

The general principle: don't manage long-lived keys. Both clouds have a way to give a workload a short-lived token without you ever seeing a secret. Use it. The era of "download a JSON credential file" is over.

Anti-pattern: managed-identity JSON key files in Azure. A managed identity's "secret" is the IMDS endpoint, not a key. If you find yourself copying a credential from a managed identity into a config file, you're doing it wrong.
