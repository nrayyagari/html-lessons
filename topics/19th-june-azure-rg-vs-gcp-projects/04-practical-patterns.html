Lesson 04 · Practice

Naming, tagging, and the layout that doesn't bite you later

The single biggest predictor of cloud-bill regret is the group/project layout you set on day one. Get it right and you can answer "what does this cost" in two clicks. Get it wrong and you spend a year untangling it.

Naming conventions

Both clouds enforce different rules (covered in lesson 1). The real question is: what's the convention you commit to as a team?

Azure RG naming

<app>-<env>-<region>-rg

Examples: payments-prod-eus-rg, search-staging-wus2-rg, data-shared-neu-rg.

Region in the name is helpful because a single subscription can host resources in many regions, and the RG is the unit of deletion — "delete the staging RG in westus2" is unambiguous.

GCP project naming

<app>-<env> (project ID has dashes; project name can have spaces for display)

Examples: payments-prod, search-staging, data-shared.

Region is not in the project name because projects are inherently multi-region. Region lives at the resource level (the bucket, the VM, the cluster).

The reason the two conventions differ: in Azure, the RG is often used to co-locate resources of a single application ("everything for payments lives in payments-prod-eus-rg, even if some of it crosses regions"). In GCP, the project usually co-locates a single application across regions naturally ("payments-prod has us-east1, europe-west1, and asia-northeast1 resources").

One rule that doesn't change. Whatever convention you pick, encode it in your Infrastructure-as-Code as a validation, not just a doc. A naming convention that isn't checked by terraform validate or a CI lint will silently drift within a quarter.

Tagging: the second axis of organization

Tags (Azure) and labels (GCP) are key-value pairs on every resource. They don't affect hierarchy or IAM, but they make filtering and reporting possible across the hierarchy.

# Azure: tags are a flat object on every resource
tags = {
  Environment = "prod"
  Team        = "payments"
  CostCenter  = "10042"
  Owner       = "alice@example.com"
  DataClass   = "pci"
}

# GCP: labels are the same shape
labels = {
  environment = "prod"
  team        = "payments"
  cost-center = "10042"
  owner       = "alice@example.com"
  data-class  = "pci"
}

Both clouds support cost-allocation reports that roll up by tag key. The ones that matter most:

TagUsed for
environment / envFilter cost reports by prod / stage / dev
team / ownerRoute alerts to the right Slack channel or pager
cost-centerChargeback to a finance code
data-class / complianceIdentify resources in scope for PCI / HIPAA / SOC2 audits
ttl / expires-afterSandboxes and dev environments with an auto-cleanup

Policy enforcement

Tags and naming conventions are only as good as their enforcement. Both clouds have a policy layer that can refuse non-compliant resources at creation time:

Azure Policy

Assignments with effect Deny / Audit / Append. Initiatives group related policies. Examples: "Allowed locations", "Required tags", "SKU restrictions".

Scope: MG, subscription, RG. Inherited downward.

GCP Org Policy

Constraints with deny / allow enforcement. Examples: constraints/compute.disableInternetNetworkEndpointGroup, constraints/iam.allowedPolicyMemberDomains.

Scope: org, folder, project. Inherited downward.

The two systems are different in detail but solve the same problem. The lesson is the same: enforce the rule at the boundary, not after the fact.

When to split a group/project

The most common "should I split this?" question, answered:

Split when…

  • Different access teams
  • Different cost owners / chargeback codes
  • Different lifecycle (sandbox vs prod)
  • Different compliance boundary (PCI vs non-PCI)
  • Resource count > 800 in one group (limits & UI slowdown)

Don't split just for…

  • Visual organization (use tags/label)
  • Region separation (resources can be in different regions within one RG)
  • "Cleanliness" (one VM is fine to live in the same RG as 50 others)

Migration: from one cloud to the other

The most common reasons teams move between Azure and GCP:

  • Acquisition — your company bought a company on the other cloud.
  • Best-of-breed — GCP's data products (BigQuery, Dataflow) are best-in-class; Azure's enterprise integration is best-in-class.
  • Cost — one cloud has better pricing for your workload shape.

The pattern that actually works:

1. Map the topology, not the resources

Don't try to map "this VM to that VM." Map the groups: what are your top-level containers, and what lives in each? You'll have the same grouping problem on the other side; it just looks different.

2. Build the new hierarchy first

Stand up the new MG tree / folder tree before you migrate anything. Validate the IAM model. Get comfortable with the cost reports. Then start moving.

3. Migrate per-group, with rollback

Pick the lowest-risk group first (internal tool, non-customer-facing). Migrate it end-to-end. Cut over. Roll back if anything breaks. Repeat.

4. Don't try to do identity federation day 1

Cross-cloud identity federation (Azure AD ↔ Google Workspace) is a real project, but it's a separate project from migration. Do it after the workloads are stable.

The single biggest migration mistake. "Let's just replicate the existing structure on the new cloud" — without asking whether the old structure was right in the first place. Most group structures have accreted over years. Use the migration as a chance to fix them.
