Lesson 01 · The honest answer

Is Go syntax hard?

Short answer: no. Longer answer: the syntax is deliberately small — fewer keywords than most languages you have used — and most of what people call "hard" about Go is the intentional omission of familiar features, not added complexity.

You have probably seen people online call Go syntax "weird" or "hard" or "missing things". Most of those comments are about expectations, not the language. Go was designed in 2007 at Google specifically because the alternatives were felt to be too complex for large teams writing large backends. The conservativeness is the feature.

What people mean when they say "hard"

Let me list the things that surprise newcomers. None of them is the syntax — they are the absences.

"Where are the classes?"

Go has no class keyword. You attach methods to any type via a receiver. The behavior is class-like, the syntax isn't. This is a feature once you internalize it — you can give methods to types you don't own (including primitives).

"Where is the inheritance?"

There isn't one. Composition only. You embed a struct inside another struct; the inner type's methods are promoted to the outer. This is the same as the existing structs & pointers series covered. It feels limiting for a week, then liberating forever.

"Where are the generics?"

Go 1.18 (2022) added them. func Map[T any](s []T, f func(T) T) []T works. The syntax is deliberately minimal — no variance, no higher-kinded types, no monads.

"What about exceptions?"

No exceptions. Errors are values: func foo() (T, error). You handle them at the call site. People who came from Python or Java find this verbose; people who came from C find it honest. The verbosity is the cost of explicitness.

Every "weird" thing in Go is an absence, not an addition. If you find yourself writing Go and wishing the language had a feature you know from elsewhere, check whether the language has a deliberate alternative. It usually does. The frustration is the adjustment, not the language.

The whole grammar, by example

Here's a complete Go file. Every keyword in the language appears in it. (Yes, really.)

package main

import (
    "fmt"
    "errors"
)

type Counter struct {
    n int
}

func (c *Counter) Inc() // pointer receiver — mutates
    { c.n++ }

func New(start int) *Counter {
    if start < 0 {
        return nil, errors.New("negative start")
    }
    return &Counter{n: start}, nil
}

func main() {
    c, err := New(0)
    if err != nil { return }
    for i := 0; i < 3; i++ {
        c.Inc()
    }
    fmt.Println(c.n)  // 3
}

Keywords used: package, import, type, struct, func, if, else, return, for, nil. That's 10. The full keyword list has 25 entries. You can learn the entire grammar in a weekend.

Go picks one way to do each of these and makes it the default. No if err != nil debate about which error-handling library; the language decides. No formatting war; gofmt decides. This is the part people love or hate.

The honest case for "it's hard"

There are a few things in Go that are actually a bit hard, especially coming from other languages:

  1. Pointers without pointer arithmetic. Covered in lessons 3-4 of this series.
  2. Interfaces are implicit. A type satisfies an interface by having the methods; there's no implements keyword. The compiler catches the broken case, but you can't tell from reading a type that it satisfies a given interface without looking at the methods.
  3. Error handling verbosity. if err != nil on every call site. It gets old. Linters help (you can ignore them).
  4. No operator overloading. You can't make + mean something for your custom numeric type. The maintainers think this is a feature; some users think it's a loss.

But notice: the things on this list are absences. The language is small on purpose. Once you know what it has, you know the whole language. That's the trade-off Go makes.
