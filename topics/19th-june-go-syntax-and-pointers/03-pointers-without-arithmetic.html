Lesson 03 · Pointers

Pointers without arithmetic

Go pointers are addresses you can read through and pass around. You can't add to them, can't subtract, can't index them. This one restriction removes the entire class of bugs C and C++ have spent decades teaching us about, and the language pays you back by giving you the parts of pointers that actually matter.

If you have used C or C++: forget everything about pointer arithmetic. If you have used Java or Python: you already think about references; pointers in Go are like references with one extra piece of syntax (*). If you have used Rust: Go pointers are non-owning, non-borrow-checked references; no lifetimes, no move semantics.

The two operators

There are exactly two pointer operators. Memorize them.

OperatorTypeEffect
&xaddress-oftakes the address of x; produces a *T
*pdereferencefollows the address in p; produces the value T behind it

That's it. No +, no -, no [], no -> (use . on any pointer, Go auto-dereferences for you — covered in the pointers series).

What you can do with a Go pointer

x := 42
p := &x              // p is *int, points at x
fmt.Println(*p)   // 42 — read through the pointer
*p = 100             // write through the pointer
fmt.Println(x)      // 100 — same memory

That's the entire pointer surface. The only things you can do with a *T:

Read the pointed-to value (*p)

Returns a copy of the value. You can also use the shorthand p.Field, which Go auto-dereferences for you.

Write through the pointer (*p = …)

The change is visible through every other reference to that same address.

Pass the pointer around

Function parameters, slice elements, map values, struct fields — all of them can hold *T. The receiver of a method can be *T.

Compare to nil

p == nil is true iff the pointer is the zero value. A nil pointer is the only "invalid" pointer — dereferencing it panics.

What you cannot do

Removing these is the entire point of Go's pointer design.

Pointer arithmetic

p := &x
p++            // ✗ compile error
p = p + 1      // ✗ compile error
p[0]          // ✗ compile error

You can't move a pointer through memory. If you want to walk a slice, walk the slice (for i, v := range s). The slice itself knows its bounds; you don't.

Pointer-to-pointer (mostly)

var pp **int  // legal but a smell

Possible, but almost always a sign of confused design. If you think you need a pointer to a pointer, you probably need a struct.

The entire C++ "iterator invalidation" problem class doesn't exist in Go. Slices and maps have value semantics (the header is copied, the backing array is shared); you can't get a pointer to a slice element and then resize the slice and have the pointer dangle — because the slice header is the only thing you'd be moving, and pointers to slice elements point at the backing array, which is stable. This is a giant category of bug that simply doesn't happen.

Garbage collection is the unspoken part

Go pointers are safe the way Java references are safe. You don't free them. You don't pair malloc with free. You don't track ownership. The garbage collector (a concurrent, tri-color mark-sweep collector since Go 1.5) figures out when nothing points at the value anymore and reclaims the memory.

What this means in practice:

  • You can return a pointer to a local variable — Go's escape analysis moves the value to the heap automatically.
  • You can share pointers across goroutines — the runtime handles synchronization of the runtime (you still need a mutex for your data).
  • You cannot use uintptr to do arithmetic on a pointer address and expect it to be safe — the GC may have moved the object. (There's a unsafe package for this; use it carefully.)

The "is it nil?" dance

A nil pointer is the closest thing Go has to a "null" object. The convention for handling it:

func UserByID(id int) (*User, error) {
    // ...
    if !found {
        return nil, ErrNotFound      // nil + error
    }
    return &u, nil                          // non-nil + nil error
}

// caller
u, err := UserByID(42)
if err != nil { return err }
if u == nil { /* defensive — usually unreachable */ }
fmt.Println(u.Name)
The defensive u == nil check is usually unreachable. The convention is "if err is nil, the other return values are usable." Linters like errcheck and staticcheck can verify this, but the human convention is: always check err first; never re-check the values for nil. The two are tied together.

What pointers are for in Go

Three legitimate reasons to use a pointer:

Mutation

The function needs to change something the caller can see. func (c *Counter) Inc() { c.n++ }. Without the pointer, the receiver is a copy and the change is lost.

Performance

The struct is large. Passing a 1KB struct by value copies 1KB. Passing a pointer copies 8 bytes. (For most purposes this doesn't matter; the Go compiler is good at not copying what it doesn't have to.)

Identity / nil-ability

You need to represent "no value" distinctly from "zero value." A *User can be nil (no user found); a User is always a User.

If your reason isn't one of these, you probably don't need a pointer. Pass by value, return by value. The simpler code is the correct code.
