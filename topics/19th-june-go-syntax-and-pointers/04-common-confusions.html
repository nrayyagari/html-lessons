Lesson 04 · The gotchas

Common confusions, demystified

Six things that bite everyone at least once. Read this once before writing production Go and you'll save yourself hours of debugging later.

1. Range loop variable capture

This one bites every Go developer once. The loop variable is reused across iterations, so capturing it in a closure captures the same variable each time.

var funcs []func()
for _, v := range []int{1, 2, 3} {
    funcs = append(funcs, func() { fmt.Println(v) })
}
for _, f := range funcs { f() }   // prints 3, 3, 3 — not 1, 2, 3!

The fix in pre-1.22 Go: shadow the variable inside the loop.

for _, v := range []int{1, 2, 3} {
    v := v                          // new variable per iteration
    funcs = append(funcs, func() { fmt.Println(v) })
}

Go 1.22 (Feb 2024) fixed this — the loop variable is now per-iteration by default. If you're on 1.22+, this gotcha is gone. If you support older Go, shadow.

2. Shadowing with :=

:= declares a new variable in the current scope if at least one of the names is new. It doesn't fail if some of the names already exist in an outer scope — it silently shadows them.

var x int = 1
if compute() {
    x := 2            // new x, scoped to this if-block
    fmt.Println(x)   // 2
}
fmt.Println(x)       // 1

This is fine when intentional. It bites when you meant to assign to the outer x but accidentally wrote := in a nested scope. Tools like shadow (from golang.org/x/tools) and govet flag this.

Rule of thumb. Use = for assignment, := for declaration. If you find yourself writing x := something in a place where x already exists in an outer scope, you almost certainly meant =.

3. Nil map vs empty map

You can read from a nil map (returns the zero value). You cannot write to a nil map (it panics at runtime).

var m map[string]int     // m == nil
fmt.Println(m["x"])  // 0 — read works, returns zero value
m["x"] = 1           // PANIC: assignment to entry in nil map

m := map[string]int{}   // explicit empty map — write works

The same rule applies to nil slices: append works, index assignment panics. Both are the "zero value is usable, but only in a read-only way" pattern.

4. Slices share the backing array

A slice is a header: {ptr, len, cap}. When you append and the capacity is exceeded, Go allocates a new backing array and the header points at it. Until then, append mutates the original array.

a := []int{1, 2, 3}
b := a[:2]               // b shares the same backing array as a
b[0] = 99
fmt.Println(a)            // [99 2 3] — a is affected

The "gotcha" cases:

Surprising

a := []int{1,2,3}
b := a
b[0] = 99     // a is changed

Explicit copy

a := []int{1,2,3}
b := append([]int{}, a...)
b[0] = 99     // a is unchanged

5. Goroutine leaks

Unlike threads, goroutines have no built-in way to die. If you go doSomething(ch) and the receiver of ch stops reading, the goroutine blocks forever — and so does its memory.

for _, item := range items {
    go func() {
        process(item)        // `item` is shared (pre-1.22) — and we have no way to cancel
    }()
}
// nothing here waits for the goroutines; if process is slow, they pile up

Fix: use a context.

ctx, cancel := context.WithCancel(context.Background())
defer cancel()

for _, item := range items {
    go func() {
        select {
        case <-ctx.Done(): return
        default:         process(item)
        }
    }()
}

Every goroutine should have a path to exit. Linters like go vet and staticcheck can catch some patterns.

6. The "interfaces are nil" trap

An interface value is a pair: (type, value). A typed nil pointer wrapped in an interface is not a nil interface. This is the single most common "why does my nil check fail" question in Go.

type Counter struct{ n int }
func *Counter Inc() { c.n++ }

var c *Counter = nil          // c is a nil *Counter
var i Incr = c                // i is (type=*Counter, value=nil) — not a nil interface!
fmt.Println(i == nil)          // false
i.Inc()                      // panic: nil pointer dereference

The fix: be explicit about the nil check at the boundary where you wrap the concrete value in the interface.

var c *Counter = nil
if c == nil {
    // return nil interface, not interface-wrapping-nil
    return nil
}
return c                       // interface holds (type=*Counter, value=&counter{})
The full rule. An interface is nil iff both the type and the value are nil. A typed nil (var p *T = nil; var i I = p) is non-nil because the type slot is filled. This is a design quirk, not a bug, but it catches everyone once.
