Lesson 01 · The model

cert-manager: who owns your TLS certs?

Three CRDs and a controller. You describe the certificate you want; cert-manager makes it exist, renews it before expiry, and stores the private key in a Secret. That's the whole shape.

TLS certificates expire. If you manage them by hand — buying from a CA, uploading to a Secret, restarting the Ingress — you will miss a renewal and your API will go dark at 03:00 on a Sunday. cert-manager is a Kubernetes controller that treats certificates as a declarative resource: you say what, the controller figures out how and when.

The three resources you write

  1. Issuer / ClusterIssuer — a named authority that can sign certs (Let's Encrypt, an internal CA, Vault).
  2. Certificate — the cert you actually want: which Issuer, which DNS names, which Secret to write to.
  3. (Secret) — standard k8s object that cert-manager writes the key+cert into, ready for Ingress to mount.

Everything else — ACME challenges, renewal, private key generation, rotation — is the controller's job.

How the controller reacts to a Certificate

You apply a Certificate manifest

The controller sees a new resource. It inspects spec.issuerRef to find which Issuer to use and spec.dnsNames to know what to sign for.

Self-check + private key generation

If a matching Secret doesn't yet have a usable cert, the controller generates an RSA / ECDSA private key locally — it never leaves the cluster.

Challenge (for ACME issuers)

For a public CA like Let's Encrypt, the controller proves it controls the domain: HTTP-01 (serve a token at http://<domain>/.well-known/acme-challenge/<token>) or DNS-01 (create a TXT record). Both can be solved automatically with the right ingress / dns01 solver config.

Certificate issued, written to the Secret

The signed cert is base64-encoded and stored in Secret.data tls.crt / tls.key, exactly where Ingress / Gateway expects it.

Renewal happens automatically

cert-manager watches every cert it issued and renews ~30 days before expiry. You don't have to do anything — the controller runs the challenge flow again, gets a new cert, updates the Secret.

The resource hierarchy at a glance

Issuer / ClusterIssuer how to sign certs · account ref, server URL, solvers
Certificate what you want · dnsNames, duration, secretName
tls.crt / tls.key where it lands · a normal k8s Secret
Ingress / Gateway reads the Secret
Issuer is namespaced. ClusterIssuer is not. Use a ClusterIssuer for certs you'll issue across many namespaces (the common case). Use a plain Issuer when you want to scope a single team's signing authority to their namespace.

A complete example: HTTPS for an Ingress, end to end

Assume cert-manager is already installed (one-line kubectl apply from cert-manager.io/docs/installation) and you have a ClusterIssuer named letsencrypt-prod.

1. The Issuer

# cluster-issuer.yaml
apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata:
  name: letsencrypt-prod
spec:
  acme:
    server: https://acme-v02.api.letsencrypt.org/directory
    email: ops@example.com
    privateKeySecretRef:
      name: letsencrypt-prod-account
    solvers:
      # HTTP-01: prove domain control by serving a token over HTTP
      - http01:
          ingress:
            class: nginx

2. The Certificate

# cert.yaml
apiVersion: cert-manager.io/v1
kind: Certificate
metadata:
  name: api-example-com
  namespace: default
spec:
  secretName: api-example-com-tls   # the Secret that gets written
  issuerRef:
    name: letsencrypt-prod
    kind: ClusterIssuer
    group: cert-manager.io
  dnsNames:
    - api.example.com

3. The Ingress that uses the Secret

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: api
  namespace: default
  annotations:
    # Optional shortcut: the cert-manager Ingress shim can create
    # the Certificate FOR you when this annotation is present.
    cert-manager.io/cluster-issuer: letsencrypt-prod
spec:
  ingressClassName: nginx
  tls:
    - hosts: [api.example.com]
      secretName: api-example-com-tls
  rules:
    - host: api.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: api
                port:
                  number: 8080

With the annotation on the Ingress, you don't even need the explicit Certificate manifest — the ingress-shim creates one for you, namespaced and matched to the spec.tls[].secretName. For most teams this is the only flow they use.

The ACME HTTP-01 challenge, in pictures

1. You apply a Certificate for api.example.com
2. cert-manager asks Let's Encrypt for a challenge "prove you control api.example.com"
3. cert-manager creates a temporary Ingress route http://api.example.com/.well-known/acme-challenge/<token> → answers the challenge
4. Let's Encrypt hits the URL, verifies the token
5. Let's Encrypt issues the cert; cert-manager stores it in the Secret
6. ~60 days later: same flow runs again, transparently

The whole dance usually takes under a minute. The only human input was the initial kubectl apply.

Common gotchas

Rate limits. Let's Encrypt has hard rate limits — 50 certs per registered domain per week. Use the staging issuer (acme-staging-v02.api.letsencrypt.org/directory) for testing — it has much higher limits and intentionally issues untrusted certs, so you can iterate without burning your production quota.
HTTP-01 needs port 80 reachable from the internet. If the cluster is firewalled or behind a private LB, HTTP-01 won't work. Either open port 80 or switch to DNS-01, which only needs the ability to create a TXT record on your DNS provider.
Private CA = a different Issuer kind. For internal services you don't want on the public internet, point cert-manager at your internal CA (Vault, step-ca, CFSSL, even a shell-script CA). The resource shape is the same; only the Issuer's spec.ca / spec.vault block changes.
Troubleshooting one-liner. kubectl describe certificate api-example-com -n default shows the current state and events with the failure reason (DNS not pointing here, solver config wrong, rate-limited, …). Follow with kubectl describe challenge and kubectl describe order for the ACME-level detail.
