Lesson 02 · Challenges

HTTP-01: how the proof of control actually works

When you issue a cert from Let's Encrypt, the CA has to be sure you actually own the domain. HTTP-01 is the simplest way: the CA hits http://your-domain/.well-known/acme-challenge/<token> and expects a specific string back. cert-manager makes that happen automatically — here's how.

ACME is the protocol Let's Encrypt speaks. Every issuance is a challenge-response: the CA gives you a token, you prove you control the domain by serving a specific response, and only then will the CA sign a cert. The "how" of that proof is a challenge type. HTTP-01 is the most common one and the easiest to reason about.

The lifecycle of an HTTP-01 challenge

1. cert-manager creates a Challenge resource

Triggered by the Certificate. The Challenge carries a token (random string), a key authorization (a hash of the token plus your ACME account key), and the URL the CA will hit.

2. The HTTP-01 solver sees the Challenge

cert-manager's built-in HTTP-01 solver creates a self-check: it stands up a temporary pod or Ingress rule inside the cluster that serves the key authorization at http://<domain>/.well-known/acme-challenge/<token>.

3. Let's Encrypt hits the URL from the public internet

LE does its own check from its infrastructure — not from inside the cluster. The response must match the key authorization exactly.

4. The Challenge is marked as valid (or errored)

If the response matched, LE issues the cert. cert-manager stores it in the Secret, and the Challenge is cleaned up (the temporary solver route is removed).

The self-check is what makes HTTP-01 robust. Before LE is even involved, cert-manager verifies the solver route is reachable from inside the cluster. If the self-check fails, you get an error in seconds rather than waiting for LE's timeout (which is minutes).

What the solver config actually does

In the ClusterIssuer, this is the block that controls HTTP-01:

spec:
  acme:
    solvers:
      - http01:
          ingress:
            class: nginx
            name: cm-acme-http-solver     # optional — only use a dedicated Ingress
FieldEffect when set
http01.ingress.classThe challenge route is created as an Ingress of this class.
http01.ingress.nameAll challenge routes land on one existing Ingress you control — useful when the cluster has only one public-facing Ingress and you don't want cert-manager to create new ones.
http01.gatewayHTTPRoute.parentRefsSame idea, but for Gateway API (Envoy Gateway, Contour, etc.) instead of Ingress.

If you don't set anything, cert-manager creates a new temporary Ingress for each challenge, named cm-acme-http-solver-<random>. That works, but it spams your Ingress list and assumes the default Ingress class is the one LE can reach.

Two common HTTP-01 failures — and the fix.

"self-check failed: … 404"

The challenge route exists in the cluster but isn't reachable from the test pod. Almost always a default Ingress class problem — the cluster has a default class that's not the public-facing one (or vice versa). Pin class: nginx (or your class) explicitly.

"waiting for http-01 challenge propagation: …"

The challenge route works internally but LE can't reach it from the internet. The cause is almost always port 80 not exposed (firewall, internal-only LB, the Ingress controller on a NodePort that the LB doesn't forward). Switch to DNS-01 (lesson 3) if you can't open 80.

What an unhealthy Challenge actually looks like

Reproduce, then read the events — in order:

$ kubectl get challenge -n default
NAME                              STATE     AGE
api-example-com-tls-1-xxxxxxx     pending   12s

$ kubectl describe challenge api-example-com-tls-1-xxxxxxx -n default
...
Events:
  Type     Reason     Age   From                Message
  ----     ------     ----  ----                -------
  Normal   Started    14s   cert-manager        Challenge scheduled for processing
  Normal   Presented  8s    cert-manager        Presented challenge using http-01
  Warning  Failed     3s    cert-manager        Waiting for http-01 challenge
                                                 propagation: …
  Warning  Errored    2s    cert-manager        Error presenting challenge: …

Each line is a state transition. The interesting ones are Presented (the solver route is up) and the Failed/Errored line at the end (the actual cause). Reading those two alone tells you ~80% of HTTP-01 bugs.

The propagation delay is real

Even when everything is configured correctly, the challenge response can take 30-90 seconds to propagate through:

  • The new Ingress rule has to be picked up by the controller.
  • The LB has to update its routing table.
  • DNS caches have to expire if you're changing hostnames.

cert-manager retries LE's challenge checks for a few minutes before giving up — usually long enough. But during the first issuance on a brand-new cluster, expect 60-120 seconds from kubectl apply to Ready=True on the Certificate.
