DNS-01 & wildcards

When HTTP-01 isn't an option — cluster is firewalled, domain is wildcarded, or you don't want a public Ingress — DNS-01 proves control by creating a TXT record instead of serving a token.

HTTP-01 is the default because it's simple: prove you control api.example.com by serving a token on port 80 of api.example.com. But that has two structural problems:

  1. Port 80 must be reachable from the public internet. Internal clusters, air-gapped environments, clusters behind a non-internet-facing LB — all no-go.
  2. It can only prove the exact hostname. Wildcard certs (*.example.com) need a way to prove control of every possible subdomain, which HTTP-01 can't do.

DNS-01 solves both. The CA gives you a token, you prove control by creating a TXT record at _acme-challenge.api.example.com with the token as the value. The CA queries DNS, sees the record, done. No inbound network path required — you only need the ability to write DNS records, which your DNS provider's API already gives you.

How DNS-01 works end to end

1. cert-manager creates the Challenge resource

Same as HTTP-01, but the type is DNS-01 instead of HTTP-01. The DNSName is _acme-challenge.<your-domain>.

2. The DNS-01 solver calls your DNS provider's API

cert-manager has built-in "solvers" for the major DNS providers (Cloudflare, Route 53, Google Cloud DNS, Azure DNS, DigitalOcean, Linode, RFC 2136, …). The solver creates a TXT record at the challenge name, with the token as the value.

3. The CA queries DNS

From its own recursive resolvers, the CA asks "what's the TXT record for _acme-challenge.api.example.com?". The record your solver just created is the answer.

4. After the CA verifies, cert-manager cleans up the TXT record

The challenge-specific record is removed. The cert is issued and stored in the Secret.

DNS propagation time. Unlike HTTP-01 (where the solver is a local route), DNS-01 depends on the TXT record being visible to the CA's resolvers. Some providers (Cloudflare, Route 53) propagate in seconds; others (some registrars) take minutes. cert-manager retries, but if your provider is slow you may want to raise solvers[].dns01.<provider>.preCheckNameservers to a public resolver like 1.1.1.1:53.

Solver config: the four you'll actually use

Cloudflare

solvers:
  - dns01:
      cloudflare:
        apiTokenSecretRef:
          name: cloudflare-api-token
          key: api-token
        email: ops@example.com

The secret is a normal k8s Secret with one key, the value being a scoped Cloudflare API token (Zone:DNS:Edit on the zones you need).

AWS Route 53

solvers:
  - dns01:
      route53:
        region: us-east-1
        # uses the controller's IAM role — IRSA or node role

No secret — cert-manager uses the AWS SDK and assumes the controller pod has IAM permission to upsert records in your hosted zone. With IRSA (IAM Roles for Service Accounts) you scope the permission to a single zone.

Google Cloud DNS

solvers:
  - dns01:
      cloudDNS:
        project: my-gcp-project
        # uses the controller's GCP service account

RFC 2136 (any DNS server with a dynamic-update key)

solvers:
  - dns01:
      rfc2136:
        nameserver: ns1.example.com:53
        tsigSecretSecretRef:
          name: tsig-secret
          key: tsig-key

This is the "I run my own BIND / PowerDNS" option. The secret holds a TSIG key pair that authorizes cert-manager to update the zone.

Wildcard certs: the killer feature

Wildcards are why DNS-01 exists in the first place — there's no other way to issue *.example.com from Let's Encrypt. The Certificate looks almost the same:

apiVersion: cert-manager.io/v1
kind: Certificate
metadata:
  name: wildcard-example-com
  namespace: default
spec:
  secretName: wildcard-example-com-tls
  issuerRef:
    name: letsencrypt-prod
    kind: ClusterIssuer
  dnsNames:
    - "*.example.com"
    - "example.com"     # base domain — LE requires both
LE will not renew a wildcard via the same challenge if the issuer has rate limits. If you're issuing many wildcards on the same base domain, batch them into a single Certificate with multiple SANs instead of one Certificate per subdomain.

Multi-provider clusters: solver selection

cert-manager tries each solver in solvers order, in each configured selector zone, until one succeeds. This lets you handle split-horizon DNS or multi-cloud:

solvers:
  # 1. try Route 53 for *.aws.example.com
  - selector:
      dnsZones:
        - "aws.example.com"
    dns01:
      route53: { region: us-east-1 }

  # 2. fall back to Cloudflare for everything else
  - dns01:
      cloudflare:
        apiTokenSecretRef: { name: cloudflare-api-token, key: api-token }
