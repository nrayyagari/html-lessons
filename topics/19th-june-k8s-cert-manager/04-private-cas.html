Lesson 04 · Issuer types

Private CAs: Vault, step-ca, self-signed

Not every cert should come from Let's Encrypt. Internal services, mTLS between pods, and offline clusters all want certs signed by a CA you control. cert-manager supports several private-CA issuer types — here's when to use which.

Let's Encrypt is great for certs the public internet has to trust. For everything else — internal services, mTLS between pods in a service mesh, certs for an offline cluster — you want a private CA. The trade-off:

Public CA (Let's Encrypt)

Trusted by browsers and OSes by default. Free. Per-domain rate limits. Domain validation only — you can't get a cert for an internal hostname like postgres.default.svc.

Private CA (Vault, step-ca, …)

Certs for any name — including internal ones. Unlimited issuance. You must distribute the CA root to every client that needs to trust the certs. Renewal is on you (or on cert-manager).

The four issuer types you'll meet

Issuer typeBest forTrusts needed
selfSignedbootstrapping, dev, ephemeral clustersnothing — not trusted by anything
cainternal CA you control (CFSSL, step-ca, openssl)clients must trust the CA root
vaultHashiCorp Vault PKI engine in your infraVault auth + clients trust the CA
venafienterprise policy/compliance (TPP, Venafi Cloud)Venafi policy

All four use the same Certificate resource — only the spec.issuerRef.kind and the Issuer's spec block change.

Self-signed: for bootstrapping, not for browsers

Use it once, throw it away. Self-signed certs aren't trusted by anything; they're useful for the bootstrap problem ("I need a cert to talk to the Vault API to get a real cert").

apiVersion: cert-manager.io/v1
kind: Issuer
metadata:
  name: self-signed-bootstrap
  namespace: default
spec:
  selfSigned: {}

One thing self-signed is useful for: as the issuer for an intermediate CA Certificate. You sign a long-lived CA cert with self-signed, then use that CA to issue real certs. The "real" trust comes from distributing that intermediate to your clients, not from the self-signed root.

CA Issuer: the do-it-yourself option

You have a CA bundle (PEM-encoded cert + key). cert-manager uses it to sign Certificate requests locally, no network round trip, no rate limit. The tradeoff: you're responsible for the CA itself — secure storage, rotation, audit, the whole nine yards.

# 1. the CA's private key & cert live in a Secret
apiVersion: v1
kind: Secret
metadata:
  name: internal-ca-key
  namespace: cert-manager
type: Opaque
data:
  tls.crt: <base64 PEM of CA cert>
  tls.key: <base64 PEM of CA key>

# 2. the Issuer points at the Secret
apiVersion: cert-manager.io/v1
kind: Issuer
metadata:
  name: internal-ca
  namespace: default
spec:
  ca:
    secretName: internal-ca-key

For most teams, this is the right "I want internal certs but I don't want to run a full PKI" option — especially if you already have a CA from your security team, or if you generate one with step ca init (see below).

step-ca: small, friendly, well-loved

step-ca is a lightweight private CA from the makers of step. The ca Issuer type talks to it over HTTP. Setup is two commands:

# 1. initialize a CA on a small VM (or in-cluster, but a VM is the usual home)
step ca init --name "Internal CA" --dns "ca.internal.example.com" \
            --address ":443" --provisioner "admin"

# 2. point cert-manager at it
apiVersion: cert-manager.io/v1
kind: Issuer
metadata:
  name: step-ca
  namespace: default
spec:
  ca:
    url: https://ca.internal.example.com
    cABundle: <base64 PEM of the step-ca root>
    privateKeySecretRef:
      name: step-ca-key
    maxRetryDuration: 30s

step-ca's killer feature is the JWK provisioner model — you issue certs with short-lived tokens instead of long-lived CA keys. The CA private key never leaves the step-ca server. Combined with cert-manager renewing at, say, 7 days, you get very tight blast radius if any one cert or token is compromised.

Vault Issuer: when Vault is already the source of truth

If you already run HashiCorp Vault and have its PKI secrets engine configured, the Vault Issuer type just calls Vault to sign. Auth is via a Kubernetes service-account JWT bound to a Vault role.

apiVersion: cert-manager.io/v1
kind: Issuer
metadata:
  name: vault-pki
  namespace: default
spec:
  vault:
    server: https://vault.internal.example.com
    path: pki_internal/sign/example-com
    auth:
      kubernetes:
        role: cert-manager
        serviceAccountRef:
          name: cert-manager
        path: auth/kubernetes/login

Vault gives you policy control over which certs can be issued for which names, audit logs of every issuance, and easy revocation via Vault's PKI engine. For regulated environments, this is the natural choice.

Distributing the CA root to clients

Private CAs aren't trusted by default — the OS / browser / language runtime doesn't know about your CA. You have to distribute the root. Two patterns:

Inject into a sidecar / init container

For service-mesh mTLS, the sidecar (Istio, Linkerd) already ships with a CA bundle — configure it to trust yours. The application code never has to know.

Mount as a configmap on the pod

For HTTPS calls between services, mount the CA cert into /etc/ssl/certs/ via a ConfigMap and point your HTTP client at the path. Most libraries support SSL_CERT_FILE automatically.

The forgotten step. Most "cert-manager isn't working" stories end with someone realizing they issued a cert correctly, but the client doesn't trust the issuing CA. curl -v the endpoint and look for self signed certificate in certificate chain — that's the smoking gun.
