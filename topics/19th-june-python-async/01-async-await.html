Lesson 01 · Foundations

async / await

A coroutine is a function that can pause itself while it waits for something. The event loop is what runs every coroutine in the same thread, switching between them the moment one of them waits. That's the whole idea.

Asyncio is Python's answer to a specific problem: your program spends most of its time waiting — for a database, for an HTTP response, for a disk read — and during that wait, the CPU has nothing to do. Threads can help, but they're expensive and racy. Coroutines cooperate: they pause themselves with await and let the loop run something else until the wait is over.

The two things to internalize

  1. One thread, many coroutines. There is no parallelism here — no two coroutines execute Python at the same time. There is concurrency: many are in flight, but only one is making progress on the CPU at any given moment.
  2. await is the only yield point. If a coroutine never hits await, it hogs the loop. That's the cooperative part — you have to play along.

The minimum useful program

import asyncio

async def main():
    print("hello")
    await asyncio.sleep(1)
    print("world")

asyncio.run(main())

Read it bottom-up:

asyncio.run(main())

Creates a fresh event loop, runs main() to completion, then tears the loop down. This is the only top-level entry point you should ever call — never poke the loop yourself.

async def main()

Defines a coroutine function. Calling main() does not run it — it returns a coroutine object, like a promise. The loop runs the body only when the coroutine is awaited or scheduled.

await asyncio.sleep(1)

Yields control back to the loop for 1 second. While this coroutine is paused, the loop is free to run other coroutines. After 1s the loop resumes us at the next line.

Calling a coroutine without await does nothing. main() returns a coroutine object, not a result. asyncio.run is what actually executes it. This is the single most common beginner mistake.

Sequential vs concurrent: the timing payoff

Consider two coroutines that each wait one second. Run them in sequence: 2s total. Run them concurrently: 1s total.

Sequential — await a; await b

A · sleep 1s
B · sleep 1s
coroutine waiting on I/O

elapsed: ~2.0s

Concurrent — gather(a, b)

A · sleep 1s
B · sleep 1s

elapsed: ~1.0s

Both A and B are scheduled together. The loop interleaves them on the same thread — while A is sleeping, B is also sleeping; while neither is runnable, the loop is idle.

The code for the concurrent version uses asyncio.gather, which lesson 2 covers in depth. For now, just notice the shape:

import asyncio

async def slow(label, secs):
    await asyncio.sleep(secs)
    return f"{label} done"

async def main():
    # both sleep at the same time
    results = await asyncio.gather(
        slow("A", 1),
        slow("B", 1),
    )
    print(results)   # ['A done', 'B done']

asyncio.run(main())

When asyncio helps — and when it doesn't

Use asyncio

I/O-bound work

The program is mostly waiting: HTTP, database, file I/O, message queues, websockets. Each await is a chance for the loop to make progress on something else.

Avoid asyncio

CPU-bound work

The program is computing: image processing, crypto, numerical work, parsing. Asyncio adds overhead without any concurrency benefit — you want multiprocessing (separate cores) or native code via C extensions / numpy.

If you have one blocking call, you block the whole loop. A 5-second time.sleep(5) inside a coroutine freezes every other coroutine for 5 seconds. Use await asyncio.sleep(5) instead, or run the blocking call in a thread pool with asyncio.to_thread (covered in lesson 2).

The event loop in one paragraph

Picture a single-threaded todo list. The loop walks the list. For each ready coroutine, it runs it until the coroutine either finishes or hits an await on something that isn't ready. The not-ready ones go on a "wait until X" list — e.g. "wait until socket A has data" or "wait until timer B fires". The loop then runs other ready coroutines. When the OS reports that one of the waited-on things is ready, that coroutine goes back on the ready list. The loop keeps cycling until there's nothing left to do, at which point asyncio.run returns.

1thread
Ncoroutines
1CPU core busy
0locks needed

The "0 locks" is the underrated win. Because only one coroutine runs at a time and they only switch at await points, most of the threading pain — race conditions, mutexes, deadlocks — simply doesn't apply. You give up true parallelism, but you buy back your sanity.
