Lesson 02 · Patterns

Tasks, gather & timeouts

Lesson 1 left you with await and a working mental model. Now the real toolset: schedule coroutines to run concurrently, gather their results, cancel what takes too long, and push blocking code off the loop.

Three ways to start a coroutine

You can think of these as "I want this to run — eventually". They differ in when the body starts and whether the caller waits for the result.

APIWhen body startsCaller waits?Returns

The third one is the unlock. create_task schedules the coroutine to run as soon as the loop gets a chance, and gives you back a Task handle. You can keep going, schedule more, then await the task when you actually need its result.

async def main():
    t1 = asyncio.create_task(slow("A", 1))
    t2 = asyncio.create_task(slow("B", 1))

    # do other work here while A and B run in the background…

    r1 = await t1   # join point 1
    r2 = await t2   # join point 2 — but B is already done by now
    print(r1, r2)

This is "fire and join" — you kick off both coroutines, do whatever else you need to do, then collect the results. If the work in the middle is non-trivial, you save real time.

gather is the shortcut. The pattern above — create all the tasks, then await each one — is exactly what asyncio.gather(*coros) does in one line, returning the results in input order.

gather in detail

results = await asyncio.gather(
    fetch(url1),
    fetch(url2),
    fetch(url3),
)
# results == [r1, r2, r3] in order; raises on first failure

Two behaviors people get wrong:

Order is preserved, not scheduling

If you pass [A, B, C] to gather, the result list is [A's result, B's result, C's result] in that order — regardless of which one finishes first. Useful when order matters; wasteful when you only need the first or any.

The first failure cancels the rest

If A raises an exception, gather cancels B and C, then re-raises A's exception. If you want all coroutines to run to completion and inspect failures individually, pass return_exceptions=True.

# gather with return_exceptions: never raises; results include Exception objects
results = await asyncio.gather(
    fetch(url1),
    fetch(url2),
    return_exceptions=True,
)
for url, r in zip(urls, results):
    if isinstance(r, Exception):
        log(f"{url} failed: {r}")
    else:
        process(r)

Timeouts: wait_for

Real I/O has no upper bound. A "fetch" can hang forever, a database query can wait for a lock forever. You need a way to give up.

try:
    result = await asyncio.wait_for(fetch(url), timeout=2.0)
except TimeoutError:
    log(f"{url} took too long")

What wait_for actually does: it races the coroutine against a timer. If the timer wins, the coroutine is cancelled — a CancelledError is raised inside it at the next await, and wait_for re-raises asyncio.TimeoutError to the caller. Cancellation is the mechanism; timeout is the reason.

Cancellation can be lost. If the coroutine catches CancelledError and doesn't re-raise it, the task keeps running in the background — you've leaked it. Always re-raise unless you have a good reason, and finally blocks for cleanup.

Offloading blocking code: to_thread

You will inevitably need to call a blocking library from inside an async program: a sync DB driver, a CPU-heavy computation, a file read. Calling it directly blocks the loop. The fix is to run it in a thread:

# Python 3.9+
data = await asyncio.to_thread(sync_db_query, "SELECT …")

# pre-3.9 equivalent
data = await asyncio.get_event_loop().run_in_executor(
    None, sync_db_query, "SELECT …"
)

to_thread hands the callable to the default ThreadPoolExecutor, runs it on a worker thread, and awaits its return value. The loop stays responsive the whole time. For heavier CPU work, prefer ProcessPoolExecutor so you actually use multiple cores.

A realistic example: concurrent HTTP fetches with deadlines

import asyncio
import httpx            # pip install httpx

async def fetch_one(client, url):
    r = await client.get(url, timeout=5.0)
    r.raise_for_status()
    return url, len(r.content)

async def fetch_all(urls):
    async with httpx.AsyncClient() as client:
        tasks = [asyncio.create_task(fetch_one(client, u)) for u in urls]
        # gather with return_exceptions so one bad URL doesn't kill the rest
        results = await asyncio.gather(*tasks, return_exceptions=True)

    for u, r in zip(urls, results):
        if isinstance(r, Exception):
            print(f"{u}: {type(r).__name__}: {r}")
        else:
            url, size = r
            print(f"{url}: {size} bytes")

asyncio.run(fetch_all([
    "https://example.com/",
    "https://example.org/",
    "https://this-domain-does-not-exist.invalid/",
]))

What this gets you for free:

  • All three requests fire at the same tick of the loop.
  • The bad URL fails fast; the other two still complete.
  • Per-request timeouts prevent one slow server from blocking the whole batch.
  • Total elapsed time is roughly max(latencies), not sum(latencies).
