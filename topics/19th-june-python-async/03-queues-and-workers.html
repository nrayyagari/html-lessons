Lesson 03 · Coordination

Queues & producer/consumer

Most real async programs aren't "fire 5 HTTP requests and gather" — they're streams of work, processed by a pool of workers, with backpressure so a slow consumer doesn't make a fast producer OOM. That's a queue.

By now you've seen gather: a one-shot fan-out that collects when everyone's done. Production code is usually streaming — a producer keeps generating work, some number of workers keep consuming it, and the system runs forever (or until a cancellation).

The right tool is asyncio.Queue. It's a thread-safe, awaitable, bounded-by-default queue with three primitives:

CallSuspends when…Wakes when…
await q.put(item)queue is full (bounded)space is available
await q.get()queue is emptyan item is available
q.task_done()—marks the most recent get as processed

The "suspends when full" behavior is the backpressure. A fast producer can't outrun a slow consumer indefinitely — the queue fills, put blocks, and the producer naturally slows down to match.

Worker pool: the canonical pattern

import asyncio

async def worker(name, q):
    async for item in q:                # async iterator — exits when q is done
        try:
            result = await process(item)
            await save(result)
        except Exception as e:
            log(f"{name}: {item} failed: {e}")
        finally:
            q.task_done()

async def main(items):
    q = asyncio.Queue(maxsize=100)   # bounded — backpressure!

    # spin up N workers; they all block on q.get() immediately
    workers = [
        asyncio.create_task(worker(f"w{i}", q))
        for i in range(8)
    ]

    # producer: hand items to the queue, blocking if it's full
    async def produce():
        async for item in items:
            await q.put(item)

    producer = asyncio.create_task(produce())

    # wait for the producer to finish putting everything
    await producer

    # wait for the queue to drain
    await q.join()

    # tell workers to exit; the async-for on q exits when the queue stops
    for _ in range(8):
        await q.put(None)      # poison pill
    await asyncio.gather(*workers)
Bounded queue > unbounded. An unbounded queue (Queue() with no maxsize) gives you no backpressure — a fast producer will happily OOM you. Always set maxsize. The right size is "enough to smooth bursts, small enough that you can detect a stuck consumer".

The poison-pill pattern, explained

Workers async for item in q loop forever — the queue never closes on its own. To shut them down cleanly, you send None (or any sentinel) once per worker. Each worker hits the sentinel, breaks out of the loop, and returns. This is the async equivalent of "send a kill signal".

There's a subtler version using asyncio.Queue.shutdown() (Python 3.13+) that avoids the sentinel. For 3.12 and earlier, the poison pill is the standard pattern.

Cancel a worker cleanly

If a worker needs to be stopped mid-flight, cancel its task and let the cancellation propagate:

async def worker(name, q):
    try:
        async for item in q:
            await process(item)
            q.task_done()
    except asyncio.CancelledError:
        log(f"{name}: shutting down")
        raise   # always re-raise — the supervisor is waiting for it

# somewhere else, when the world is ending:
worker_task.cancel()
try:
    await worker_task
except asyncio.CancelledError:
    pass

Two non-obvious rules from lesson 2, restated because they matter more in workers:

  1. Always re-raise CancelledError — a swallowed cancellation leaks the worker.
  2. Do cleanup in finally blocks (close file handles, ack messages, flush buffers).

When to use a queue vs gather

Use gather

You have N coroutines, all started at the same point, all expected to complete (or fail) before you move on. One-shot fan-out, finite.

Use a queue + workers

Work arrives over time (a stream, a message consumer, a file reader). You want a fixed pool of workers and explicit backpressure. Runs until cancelled.
