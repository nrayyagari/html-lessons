Lesson 05 · Ecosystem

Common libraries: HTTP, files, databases

Once you can write async def, the rest of the work is knowing which library to reach for. The standard library has almost nothing async. The ecosystem has a few standouts that cover 80% of real code.

The async ecosystem is small compared to the sync one. The libraries below are the ones you'll actually use in production code. The pattern is consistent: there's an async-first library (use it), and a sync one that you wrap in asyncio.to_thread when you have to (don't reach for it by default).

HTTP: httpx

httpx is the modern HTTP client. It has a sync and an async API in one package, and the async API is the one to reach for.

import httpx

async def main():
    async with httpx.AsyncClient(
        timeout=10.0,
        follow_redirects=True,
        headers={"user-agent": "myapp/1.0"},
    ) as client:
        # one request
        r = await client.get("https://api.example.com/v1/things")
        r.raise_for_status()
        data = r.json()

        # many in parallel — same pattern as lesson 2
        urls = [f"https://api.example.com/v1/things/{i}" for i in range(100)]
        results = await asyncio.gather(
            *(client.get(u) for u in urls),
            return_exceptions=True,
        )

The AsyncClient is the right unit of reuse: it manages a connection pool, so 100 get() calls reuse the same TCP connections instead of opening 100 new ones. Always use async with AsyncClient() — the client cleans up the pool on exit.

Don't use requests in an async program. requests is blocking. Wrap it in asyncio.to_thread only if you must (legacy code), and even then prefer httpx — the API is nearly identical and the migration is mechanical.

Files: aiofiles

Python's built-in open() is blocking. aiofiles gives you an async wrapper with the same API:

import aiofiles

# read line by line without blocking the loop
async def read_lines(path):
    async with aiofiles.open(path, mode="r") as f:
        async for line in f:
            yield line

# write a big file in chunks
async def write_report(path, chunks):
    async with aiofiles.open(path, "w") as f:
        for chunk in chunks:
            await f.write(chunk)

aiofiles is a thin wrapper — under the hood it runs the blocking open / read / write in a thread pool. The savings come from not holding the loop hostage during the I/O, not from raw throughput.

PostgreSQL: asyncpg

asyncpg is the fastest PostgreSQL driver in the ecosystem and is async-native. It has its own connection pool, asyncpg.create_pool.

import asyncpg

async def main():
    pool = await asyncpg.create_pool(
        dsn="postgres://user:pass@db:5432/app",
        min_size=2,
        max_size=10,
    )

    async with pool.acquire() as conn:
        async with conn.transaction():
            rows = await conn.fetch("SELECT id, name FROM users WHERE active = $1", True)

    await pool.close()

For an ORM, SQLAlchemy 2.0 has full async support via AsyncSession — the migration is mostly s/session.execute/await session.execute/.

Redis: redis-py asyncio mode

The official redis client has an async mode that mirrors the sync API. Use redis.asyncio.Redis.

import redis.asyncio as redis

async def main():
    r = redis.Redis(host="cache", decode_responses=True)
    await r.set("key", "value", ex=60)
    val = await r.get("key")
    await r.aclose()

The wrap-in-thread fallback

Sometimes the library you need is sync only. The right move isn't "give up on async" — it's asyncio.to_thread:

# legacy library that only has a sync API
import legacy_pdf_lib

async def make_pdf_async(data):
    # run the blocking call in a worker thread
    pdf_bytes = await asyncio.to_thread(legacy_pdf_lib.render, data)
    return pdf_bytes

The loop stays responsive while the sync code runs in the default ThreadPoolExecutor. It's not free (thread context switches, ~1MB stack per thread) but for I/O-bound legacy code it's the right bridge.

Don't wrap CPU work in to_thread. The GIL still serializes Python bytecode across threads. For CPU-bound work, use ProcessPoolExecutor or a native extension (numpy, Cython, Rust via PyO3).

Testing: pytest-asyncio

The pytest-asyncio plugin lets you write async def test_… functions. Without it, pytest tries to call your coroutine and gets a coroutine object instead of a result.

# pyproject.toml
[tool.pytest.ini_options]
asyncio_mode = "auto"

# tests/test_fetch.py
import pytest

async def test_fetch_returns_200(httpx_mock):
    httpx_mock.add_response(url="https://api.example.com/", json={"ok": True})
    r = await fetch("https://api.example.com/")
    assert r.status_code == 200

async def test_fetch_raises_on_500(httpx_mock):
    httpx_mock.add_response(url="https://api.example.com/", status_code=500)
    with pytest.raises(httpx.HTTPStatusError):
        await fetch("https://api.example.com/")

The httpx_mock fixture (from pytest-httpx) intercepts network calls so your tests run offline and are deterministic.
