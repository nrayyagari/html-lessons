Lesson 03 · Methods

Value vs Pointer Receivers

A method is just a function with a receiver baked into the call. Whether the receiver is T or *T decides whether the call sees a copy or the original — and whether the method is in T's interface set, or only in *T's.

type Counter struct { n int }

// value receiver — sees a copy
func (c Counter) Value() int { return c.n }

// pointer receiver — sees the original
func (c *Counter) Inc() { c.n++ }

Calling these methods:

c := Counter{n: 0}
c.Value()   // Counter.Value
c.Inc()     // Go rewrites to (&c).Inc() automatically
c.Value()   // 1

When the receiver is *Counter, Go takes the address of c for you — provided c is addressable. Local variables, slice elements, and struct fields are addressable; map values and the result of a function call are not.

Mutation: only the pointer receiver can

value receiver

func (c Counter) Inc() {
    c.n++    // mutates the COPY
}

main:
  c := Counter{n:0}
  c.Inc()        // copy of c goes in
  c.n            // still 0

pointer receiver

func (c *Counter) Inc() {
    c.n++    // mutates the real one
}

main:
  c := Counter{n:0}
  c.Inc()        // Go does (&c).Inc()
  c.n            // 1

This is the "go through the arrow" pattern from lesson 2, baked into the call syntax.

The "but it worked" trap with auto-addressing

A common surprise: this does compile, and does mutate:

func (c *Counter) Reset() { c.n = 0 }

var c Counter      // addressable local
c.Reset()            // OK: Go takes &c

// vs.

func makeCounter() Counter { return Counter{} }
makeCounter().Reset() // ERROR: cannot take address of makeCounter()
Addressability rule. You can call a pointer-receiver method on a value only if that value is addressable. The result of a function call is not addressable, so foo().Bar() works for value receivers only.

Method sets: the interface rule

The set of methods on a type is called its method set. It is the most common reason people hit "method has pointer receiver" interface errors.

ReceiverMethod set of TMethod set of *T
value Tincludes itincludes it (Go auto-addresses)
pointer *Texcludes itincludes it

So if an interface demands a pointer-receiver method, only a *T can satisfy it. A value T cannot — even though the call t.PointerMethod() would work in isolation.

type Incr interface { Inc() }
var _ Incr = Counter{}      // ✗ cannot use Counter (value) as Incr
                                //   Inc has pointer receiver
var _ Incr = &Counter{}      // ✓

How to choose — the practical rule

The Go community's rule of thumb (from CodeReviewComments):

  1. The method must mutate the receiver? Pointer.
  2. The receiver is large? Pointer — avoid the copy.
  3. The receiver contains a sync.Mutex or similar? Pointer — copying the mutex is a bug.
  4. Any other method on the type uses a pointer receiver? Use pointer receivers for all of them — for consistency, and to keep the method sets compatible.
  5. Otherwise, value receiver is fine.
Consistency wins. A type with three pointer-receiver methods and one value-receiver method is a smell. The value one is probably a bug. Pick one and stick with it.
