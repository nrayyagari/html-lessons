Lesson 04 · Composition

Embedding & Method Promotion

Go's answer to inheritance: a struct with no field name automatically exposes that type's fields and methods at the outer level. There's no extends — just types living inside types.

An embedded field is a field declared with a type but no name:

type Point struct { X, Y int }
func (p *Point) Move(dx, dy int) { p.X += dx; p.Y += dy }

type Sprite struct {
    Point          // ← embedded, no name
    Name string
}

That single nameless field is enough to "promote" Point.X, Point.Y, and (*Point).Move up to Sprite:

s := Sprite{Point: Point{X: 1, Y: 2}, Name: "hero"}
fmt.Println(s.X, s.Y)   // 1 2 — promoted fields
s.Move(3, 4)         // promoted method, s.Point.X is now 4
s.Point.Move(3, 4)   // same thing, fully qualified

Memory: the embedded struct is right there

Embedding is not a pointer indirection by default. The inner struct lives inside the outer one, in the order the fields were declared.

Stack · main
frame for main
s Sprite
Point.X = 1
Point.Y = 2
Name = "hero"

The total size of Sprite is the size of Point plus the size of string, plus any padding. There is no hidden pointer to Point.

Embedding a pointer

You can embed *Point instead, and the same promotion happens — but now mutation through the promoted method is the only way to mutate, because the outer struct holds a pointer to a separately allocated inner one.

type Sprite struct {
    *Point          // embedded pointer
    Name string
}

s := &Sprite{Name: "hero"}    // s.Point is nil!
s.Move(1, 1)                  // panic: nil pointer dereference
s.Point = &Point{}             // init first
s.Move(1, 1)                  // OK now
Embedded pointer is a footgun. If you forget to initialize it, every promoted method call is a panic. The compiler can't help you.

Method promotion rules

Promotion is shallow, not transitive. If Sprite embeds Point and Level embeds Sprite, Level does not get Point's methods directly — only through Level.Sprite.Point.

Notice: value-embedded value is the only combination that does not get pointer-receiver methods of the inner type when the outer type is a value. The outer's method set only contains them if the outer is a pointer.

Composition over inheritance

The idiomatic shape of an embedding-heavy Go type:

type Reader interface { Read(p []byte) (int, error) }
type Logger interface { Logf(string, ...any) }

type Service struct {
    Reader  // embed an interface — small, no allocation
    Logger
    Name string
}

// Service.Read delegates to the embedded Reader; same for Logf.
func (s *Service) Read(p []byte) (int, error) {
    s.Logf("read", s.Name)   // promoted method
    return s.Reader.Read(p)  // delegated
}

Embedding an interface is a common pattern: a small field, zero bytes of state, full method promotion. io.ReadWriter in the standard library is the canonical example.

Quiz

Why does s.Move(1, 1) panic when Sprite embeds *Point and s.Point is still nil?

Sprite embeds value Point. From a value Sprite (not a pointer), which methods are reachable?
